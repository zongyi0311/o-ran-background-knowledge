# 1.0 為何代理式 AI 正在重塑網路運維

電信產業正進入關鍵轉折點。面對快速增加的營運壓力，從傳統的人力與半自動化模式，必須轉向更智慧、更具自主性的網路管理架構。這股「代理式 AI（Agentic AI）」潮流並非只是技術升級，而是面對未來網路環境時的必然選擇。

### 推動此轉型的核心挑戰包括：

- **複雜度不斷上升**  
  未來 6G 網路的密度、異質化、與應用多樣性會遠超過 5G。人工或集中式決策已無法管理這些複雜性，必須依靠分散式、多代理的智能模型。

- **降低 OPEX 的經濟壓力**  
  隨著基地台、邊緣節點、網路功能的規模持續擴張，沒有高自動化將無法維持成本結構。

- **單一 AI 無法完成所有任務**  
  大型意圖需要分解，單一 Agent 會成為瓶頸，因此需要多代理協同才能實現可擴展的網路智慧。

- **新興應用對跨領域協同的需求**  
  AR/VR、Metaverse 等應用需要跨應用、跨網路、跨服務的同步協作，傳統 SON 或 NMS 已無力負荷。

### 最終願景：自我合成、自我演化的網路（Self-Synthesized Network）
這類網路應能：
- 自主重建自身功能  
- 隨情境與需求演化  
- 自主調整組態與優化行為  

BubbleRAN 為此提出 **MX-AI 多代理平台**作為實現途徑。

# 2.0 MX-AI 多代理架構解析

MX-AI 是 BubbleRAN 建構的多代理智能平台，支援跨多供應商、多模型、多雲環境的網路應用開發、部署與協作。

## 核心組件如下：

| 組件 | 功能 |
|------|------|
| **MXAI Orchestrator** | 多代理系統的中央規劃者，負責任務分配與結果整合 |
| **Digital Twins** | 數位分身環境，用於安全測試代理操作前的行為 |
| **MX Hub** | 共享代理、xApps、rApps、LLM 的註冊與流通中心 |
| **MX Data** | 資料湖，提供資料交換、訂閱、分析的基礎 |
| **Agentic Toolkits & SDK** | 用於快速開發新代理的工具，包含標準化流程與模板 |

### MX-AI 能力對應產業需求：

1. **管理 5G/6G 的複雜性**（透過自動化與閉環控制）
2. **將運維從被動變為主動**（預測異常與預防性修復）
3. **大幅縮短 MTTR**（從數小時變成數分鐘）
4. **支援多模型部署**（邊緣小模型 / 雲端大模型）
5. **建立開放生態系**（代理與應用可在 MX Hub 中共享）

MX-AI 的實際部署核心，即是 **AI Fabric**。

# 3.0 AI Fabric：統一且宣告式的部署模型

多代理系統若沒有標準化部署方式，將在維護上變得極為複雜（尤其是 LLM 與工具間的協調）。  
AI Fabric 提供了解決方案：

### **AI Fabric 是可部署的最小單元，包含：**
- 一組 Agents  
- 多個 LLM（本地 / 雲端）  
- 工具與服務  
- 訊息通訊機制  
- 配置與生命週期管理  

---

## AI Fabric 的兩大核心架構：

### **1. 多代理架構（A2A Protocol）**
Fabric 內代理透過標準化 A2A 協定溝通，確保一致與高可靠性。  
必要時可透過 MCP 與外部工具對接。

### **2. 多模型架構（Multi-LLM）**
Fabric 內代理可使用不同模型：  
- 大模型用於複雜推理  
- 小模型提升效率與隱私  

AI Fabric 提供的是完整封裝的容器化運行單元。

# 4.0 代理式工作流程實務

BubbleRAN 透過三個 demo 展示 MX-AI 的完整能力：部署、開發、優化。

---

## 4.1 以意圖驅動的 AI Fabric 部署

使用者只需一份 YAML，即可定義含 orchestrator + SMO 的 Fabric。  
並可透過自然語言直接請求：

> “Deploy a new 5G network.”

系統會自動詢問必要參數（名稱、接入網路、是否部署 near-RT RIC），並完成部署。  
拆除網路同樣只需一句話：

> “Tear down the network.”

→ 展示真正的 **Intent → Action** 全生命週期管理。

---

## 4.2 自訂代理開發與整合

BubbleRAN SDK 可快速建立新代理，例如簡單的 RNG Agent。

SDK 會自動處理：
- A2A/MCP server  
- Metadata  
- Chat history  
- Workflow（如 React pattern）

開發者只需：
1. 定義 agent card  
2. 建立 agent 邏輯（graph）  
3. 撰寫工具（Python function）  
4. 打包成 container  
5. 新增至 YAML → 即可部署至 Fabric  

→ 展現易開發、高擴充性。

---

## 4.3 範例：RAN 上行功率控制的共生代理優化

此 demo 展示 MX-AI 在 live 5G OTA（over-the-air）網路中，如何自動最佳化 UL power control 的 P0_nominal。

### **Fabric 組成：**
- orchestrator  
- self-reconfiguration agent  
- **KPI_mon**（即時監測）  
- **P0_enforce**（修改 P0 參數）

---

### **操作流程：**

1. 讀取現場流量（23 Mbps），顯示 P0 = -90  
2. 使用者下指令：「最佳化 KPI」  
3. 代理分析後建議：P0 → **-100**  
4. 使用者確認  
5. 系統套用新設定  
6. 重新測試 → 吞吐量提升至 **26–28 Mbps**

此案例代表：

> **LLM 推理 + 決定論工具（deterministic tools） 的共生代理架構，是未來 RAN 自動化的最佳模式。**

# 5.0 未來展望：走向自我合成網路

BubbleRAN 在 ITU forum 中提出下一個世代網路願景：  
**Self-Synthesized Network（SSN）自我合成網路**

此網路可：
- 自主組裝功能  
- 自主修復  
- 自主演化  
- 隨需求與時間改變自己  

## SSN 的四大支柱：

- **人機介面（HMI）**  
  將可能透過 Metaverse 進行網路操作。

- **開放生態系**  
  任何模型、代理、工具都是 plug-and-play。

- **群體智慧（Collective Intelligence）**  
  多代理協同推理，形成超越單一代理的智能。

- **自治網路服務**  
  最底層的網路功能具備高度可程式化與自動化能力。
